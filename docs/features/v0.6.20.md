## FEATURE_041: Tool Output Guardrails 与 Context Overflow Protection

**Status**: Planned  
**Priority**: High  
**Category**: Enhancement

### 1. 概述

为 KodaX 增加一套“工具输出边界保护 + 大文件读取预检 + 提示词/工具契约收敛 + 运行时观测”的完整方案，解决单次 tool result 过大导致上下文暴涨的问题。

这项 feature 不是单纯调整 compaction 阈值，而是把防线前移到 tool-result 边界，避免把：

- 大文件整读
- 巨量 shell 输出
- 大型 `write/edit` diff
- 其他意外超大工具结果

直接塞回会话历史后，再被动等待 compaction 补救。

**核心目标**:
1. **单次工具结果不可直接打爆上下文** - 任意字符串型 tool result 在进入会话前都必须经过统一硬边界
2. **大文件默认不整读** - `read` 在未显式分段时优先拒绝不合理整读，并引导使用 `offset/limit`
3. **provider/model 无关** - 防护不依赖某个模型是否返回准确 usage，也不依赖 compaction 是否已触发
4. **保留可用性** - 输出被截断后仍能继续探索，必要时保留完整输出落盘路径
5. **可观测可验证** - 对超大结果、截断、拒绝整读和 token 跳变补齐日志与测试

**背景**:
- 当前 `read` 默认可整读整个文件，且没有硬性输出上限
- `bash` 正常成功路径会返回完整 stdout/stderr
- `write/edit` 大改动会返回完整 diff
- 自动 compaction 发生在迭代边界，不足以防止单轮工具结果先把会话拉爆
- 对比 `pi-mono` 和 `opencode`，KodaX 目前缺少“工具级截断 + 全局兜底”这两层保护

  - 工具级策略定义，例如 `read=head`、`bash=tail`、`diff=head+summary`

- 与现有 compaction 共存，但不再把 compaction 当作唯一防线

- 不改变权限语义，不额外引入新的 permission mode

### 3. 技术方案

#### 3.1 全局 tool-result 兜底层

在 tool result 写入会话历史前，增加统一的字符串输出截断层。

```typescript
interface ToolResultPolicy {
  maxLines: number;
  maxBytes: number;
  direction: 'head' | 'tail';
  spillToFile?: boolean;
}

function applyToolResultGuardrail(
  toolName: string,
  content: string,
): { content: string; truncated: boolean; outputPath?: string }
```

**默认建议**:
- `maxLines`: 2000
- `maxBytes`: 50 * 1024
- `read/grep/write/edit`: `head`
- `bash`: `tail`

**设计原则**:
- 这是最后一道保险，任何工具遗漏本地限制时，仍然不会把超大结果直接塞进 session
- 若工具已自行标注“已截断”，全局层只做轻量校验，不重复破坏语义

#### 3.2 `read` 预检 + 流式读取

`read` 改为“两段式”：

1. **预检**  
   - 先 `stat`
   - 如果未传 `limit` 且文件明显过大，直接返回拒绝整读的指引
   - 对极大文件、疑似生成产物、单行超长文本给出更明确建议

2. **流式读取**  
   - 不再 `fs.readFile()` 整文件后再切片
   - 按行读取，到 `limit` 或 `maxBytes` 立即停
   - 自动附带 continuation 提示

```typescript
if (!limit && stat.size > LARGE_READ_PREFLIGHT_BYTES) {
  return [
    `[Read Guardrail] File is too large for an unbounded read.`,
    `Use offset/limit to read a section, or grep to find the relevant area first.`,
  ].join('\\n');
}
```

**语义目标**:
- 手动指定 `offset/limit` 时仍然好用
- 未指定时默认安全，而不是默认整读
- 对长文件给模型清晰的“下一步怎么继续”

#### 3.3 `bash` / `write` / `edit` / `grep` 的专用策略

**`bash`**
- 正常成功路径也必须截断
- 优先保留 tail，便于保留错误和最终状态
- 如果输出被截断且允许，保存完整输出到临时文件并给出路径

**`write` / `edit`**
- 小修改仍返回可读 diff
- 大修改改为“摘要 diff + 统计 + continuation 提示”
- 必要时提供完整 diff 落盘路径，而不是把整份 diff 塞回上下文

**`grep`**
- 保留现有结果数量限制
- 增加单行长度与总字节量限制
- 当结果太多时，明确提示模型缩小 pattern 或 path

#### 3.4 提示词与工具契约更新

系统提示词与工具 schema 要主动教育模型：

- 大文件不要整读
- 先 `grep/glob` 再 `read`
- `read` 默认是分段查看工具，不是整文件导出工具
- 遇到 `[Truncated]` 或 guardrail 提示时，用 `offset/limit` 继续，不要重复读整文件
- `bash` 不是文件读取工具；日志或大输出要配合 `read/grep` 二次探索

**工具说明建议**:
- `read`: 默认最多返回 N 行 / M KB；继续请用 `offset`
- `bash`: 输出可能被截断；完整输出会保存到临时文件
- `write/edit`: 大 diff 会被摘要化显示

#### 3.5 运行时观测

为诊断类似“为什么会到 1833k tokens”补上观测：

```typescript
interface ToolResultTelemetry {
  tool: string;
  originalBytes: number;
  finalBytes: number;
  truncated: boolean;
  outputPath?: string;
}
```

**记录内容**:
- 工具原始结果字节数
- 截断后字节数
- 是否落盘
- 是否命中 `read` 预检拒绝
- 当轮 token snapshot 的跳变量

### 4. 实施步骤

1. 新增共享 truncation 工具与策略定义
2. 在 `agent.ts` 工具结果入会话前接入全局 guardrail
3. 重写 `read` 为预检 + 流式读取 + continuation 提示
4. 收紧 `bash` 正常成功路径输出
5. 收紧 `write/edit` 大 diff 输出
6. 为 `grep` 增加字节和单行长度边界
7. 更新 `registry.ts` 工具描述与 `system.ts` 提示词
8. 增加 telemetry 与高信号日志
9. 补齐单元测试、集成测试和回归测试

### 5. 验收标准

- [ ] 单次超大 `read` 不会再把整个文件直接塞进上下文
- [ ] 未传 `limit` 且文件过大时，`read` 返回明确的分段读取指引
- [ ] `bash` 正常成功路径的大输出也会被截断，而不是只在 timeout 时处理
- [ ] `write/edit` 的大型 diff 不会原样进入 session history
- [ ] 即使某个工具忘记本地截断，全局 guardrail 仍能兜住
- [ ] 截断后的结果始终给出下一步可执行的 continuation 提示
- [ ] 自动 compaction 触发前，不会再因单次 tool result 造成异常级上下文膨胀
- [ ] 新增测试覆盖 `read`、`bash`、`write/edit`、global guardrail、prompt guidance 关键路径
- [ ] 不影响现有正常的小文件读取、小规模 bash 输出和常规编辑体验

### 6. 风险与兼容性

**主要风险**:
- 截断阈值过小会影响正常调试体验
- `read` 预检过严可能让模型对中型文件也频繁“被拦住”
- `write/edit` diff 摘要化如果设计不好，可能降低 patch 可解释性

**缓解策略**:
- 将全局 guardrail 作为兜底，优先在工具级保留语义化结果
- 初始阈值选择保守值，并通过测试仓库与真实任务回放校准
- 对 `bash` 使用 tail，对 `read` 使用 head，对 diff 使用摘要，避免一刀切

### 7. 参考对比

**pi-mono**
- `read` 默认 2000 行 / 50KB，并附 continuation 提示
- 工具级 truncation 较完整

**opencode**
- `read` 有 2000 行、2000 字符/行、50KB 的硬限制
- 更关键的是有全局 tool-output truncation wrapper，能兜住遗漏工具

**KodaX 本次方案**:
- 采用 `opencode` 的“全局兜底”思路
- 结合 `pi-mono` 的“工具语义化 continuation 提示”
- 再补上 KodaX 自己所需的 `read` 预检与 telemetry
